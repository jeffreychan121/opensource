# GIT分支管理



## 1.  分支定义

- main分支
  - 主分支（github/gitlab原来叫master分支，后面改叫main分支）。
  - 在创建仓库后，main分支会被自动创建。
  - 只有项目的Maintainer有权限修改。
  - 此分支的代码，和生产环境相对应。
- release分支
  - 发布分支。
  - 在测试环境测试通过后，将代码放到对应的`release/<$version-numbber>`分支。
  - 只有项目的Maintainer有权限修改。
  - 此分支的代码，和预发环境（UAT）相对应。

- dev分支
  - 开发分支。
  - 用于开发阶段的集成、测试，包含所有开发中的功能和修复。
  - 只有项目的Maintainer有权限修改。
  - 此分支的代码，和测试环境相对应。
- feature分支
  - 每个新功能，可单独拉一个feature分支。
  - feature的内容，在开发完成并研发自测通过后，提交Merge Request，由项目负责人进行 Review，合并到dev分支。
  - developer有权限修改。
  - 此分支的代码，和研发自己（本地）的环境相对应。
- bug分支
  - 为修复bug，可拉一个单独的bug分支
  - bug分支和feature分支类似。在开发完成并研发自测通过后，提交Merge Request，由项目负责人进行 Review，合并到dev分支。
  - developer有权限修改。
  - 此分支的代码，和研发自己（本地）的环境相对应。


## 2. 分支命名约定

- `main`：生产环境分支，分支名称固定为“main”。
- `release/<version-number>`：用于发布准备的分支。格式为 “feature/” + 版本号，如：
  - release/1.2.0			为 1.2.0 版本发布准备的分支
- `dev`：开发分支，分支名称固定为“dev”。
- `feature/<feature-name>`：新功能分支。名称以“feature/”开头，如：
  - feature/zhangsan-add-change-password-function    张三为“增加修改密码”功能拉的分支    
  - feature/zhangsan-sprint3                          张三为sprint3拉的分支（不太好的做法）
  - feature/zhangsan-requirement-201        张三为 201 这个需求拉的分支

- `bugfix/<bug-name>`：bug修复分支。名称以“bugfix/”开头，如：
  - bugfix/bug-1001		为修复 1001 这个 bug 拉的分支



## 3. 开发阶段

- 首先从main分支，创建dev分支。这个动作是一次性的。
- 从`dev`分支创建一个新feature功能或bug修复分支。
- 在feature功能或bug修复分支上进行开发和测试。使用的是研发自己的本地测试环境。
- 开发完成后，提交并合并（Merge Request）到`dev`分支。
- 项目负责人对MR进行Review，通过后，将代码合并到dev分支。
- 将`dev`分支的内容，部署到测试环境，进行集成测试。

## 4. 发布阶段

- `dev`分支上的代码，部署到测试环境，进行集成测试。当测试通过后，准备好发布时，创建一个`release`分支（如`release/1.0.0`）。

- 将`release`分支的内容，部署到预发环境，进行测试。
  - 如果有测试有bug，则研发需要在feature分支上修改bug，走开发测试流程。

- 预发环境测试通过后，将`release`分支合并到`main`分支。
- 给`main`分支打上对应的标签（tag），如`1.0.0`。
- 将该tag的代码，部署到生产环境。注意，必须使用tag的代码，而不是main分支的代码，去部署生产环境。

## 5. tag管理

每次的版本发布，必须打相应的标签（tag）。标签从`main`分支打。这有助于回溯和查看每个版本的状态。

## 6. 代码审查和合并请求

在合并代码到`dev`或`main`分支前，通过Pull Request或Merge Request进行代码审查。



# 发布流程

## 1. 确定版本号

- 版本号需遵循语义化版本号格式，三段式：`MAJOR.MINOR.PATCH`。

  - **MAJOR**：有重大更改，可能包含不兼容的 API 变更。

  - **MINOR**：添加新功能，不影响向后兼容。

  - **PATCH**：修复 bug，不影响向后兼容。

  参见：https://semver.org/lang/zh-CN/

## 2. 更新`CHANGELOG.md`

`CHANGELOG.md`文件需持续维护，所有历史版本的变更记录，都放在此文件中。示例：

```markdown
## [1.6.0] - 2024-08-19
### 新增功能
- 【销售/采购】新版本企业微信审批流
  - 修改为卡片形式展示
  - 同时支持旧版本的点击进入应用审批，在当前企业微信页面审批
- 【销售】快速下单和报价单下单时显示促销活动提示
- 【客户】所有客户信用额度使用情况统计页面（菜单路径：销售/订单/信用账户）

### 修改功能
- 【销售】免税的订单限制不能生成正式的税票
- 【采购】雅致靓公司针对由YZL采购单生成的由雅致靓代采购的询价单据，采购员不能添加新的产品行
- 【采购】雅致靓公司出库单“验证”按钮替代为“添加波次”按钮

### 已知问题
- 【销售/采购】在企业微信点击卡片中“审批同意/拒绝”按钮，状态更新延迟2秒左右，待优化。

## [1.5.0] - 2024-08-12
### 新增功能
- 【销售】折扣修改为两种： 固定金额（含税）、全局折扣（以含税价折扣） 
  * 解决系统原有折扣税计算逻辑和现实不一致问题
  * 折扣最终结果只有每行折扣比例，如果是需要固定金额，会折算成百分比
  * 支持从单个明细行上面设置折扣
- 【客户】创建客户，个人和公司进行切换，公司及公司带出的相关字段数据清空
- 【采购】根据询价明细创建询价单操作，可根据供应商和采购订单号进行选择
- 【收银】收银员列表页面，默认搜索只显示当天订单

### 修改功能
- 【客户】联系人表单页面，销售与采购标签页中销售和采购内容隐藏
- 【库存】库存/报告/库存菜单，列表视图“发信”改成“出货”
- 【收银】收银员收款界面，只有会计有修改“收款日期”权限
- 【会计】部分付款，剩余款项不支付，可直接对账变成全额支付

### 修复问题
- 【销售】销售员折扣在权限范围内，叠加折扣无法触发审批流
- 【销售】打印报价单，提货单显示样式问题

### 移除
- 【销售/采购】销售、采购订单列表视图隐藏“统计视图”功能

### 已知问题
- 【销售】折扣计算精度，导致总计金额有一定误差
```

## 3. 构建和测试构建项目，生成发布包

## 4. 测试通过后，打标签

标签名字就是版本号，不带“V”，如`1.1.0`。

## 5. 创建release

在gitlab中，选择对应的tag，然后基于此tag创建release，填写release note。

release note可以直接使用CHANGELOG.md的内容。

注意，这个release不是release分支，而是指产品的一个“release(发布)”。

## 6. 部署上线

由运维人员将产品部署上线。

上线后，需由测试人员进行测试。如果测试未通过，则回退系统。
