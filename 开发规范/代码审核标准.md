# 代码 Review 标准范本（Code Review Checklist）

## 一、代码结构（Code Structure）

| 检查项         | 说明                                      |
| -------------- | ----------------------------------------- |
| 模块职责清晰   | 单一职责原则，避免类/函数过于臃肿         |
| 目录结构合理   | 按功能或领域模块划分，便于维护            |
| 是否有重复代码 | 使用函数封装、复用已有逻辑，遵循 DRY 原则 |

---

## 二、代码可读性（Readability）

| 检查项               | 说明                                         |
| -------------------- | -------------------------------------------- |
| 变量/函数/类命名规范 | 命名清晰、语义明确、无缩写或拼音（除非约定） |
| 函数长度适中         | 一般建议函数不超过 50 行，复杂函数应拆分     |
| 逻辑清晰、易于理解   | 包括条件判断、异常处理、循环逻辑等           |
| 注释得当             | 说明复杂逻辑、边界情况，避免无意义注释       |

---

## 三、编码规范（Coding Style）

| 检查项       | 说明                                                     |
| ------------ | -------------------------------------------------------- |
| 遵守语言规范 | PEP8（Python）                                           |
| 缩进一致     | 统一使用 2 或 4 空格，禁止混用空格和 Tab                 |
| 空行合理     | 逻辑区分明确，不影响阅读流畅性                           |
| 日志规范     | 日志级别（info/warning/error）使用合理，不能代替异常处理 |

---

## 四、安全性（Security）

| 检查项                    | 说明                               |
| ------------------------- | ---------------------------------- |
| 参数校验是否完善          | 尤其是用户输入、外部数据、接口请求 |
| 避免 SQL 注入/命令注入    | 使用 ORM 或参数绑定，禁止拼接 SQL  |
| 加密/解密、Token 使用安全 | 遵循加密算法标准，不暴露敏感信息   |
| 权限校验是否齐全          | 包括接口级别、数据级别等           |

---

## 五、性能与效率（Performance）

| 检查项                 | 说明                                    |
| ---------------------- | --------------------------------------- |
| 数据库查询是否优化     | 是否有 N+1 查询，是否需要加索引         |
| 算法/循环是否高效      | 是否存在多余的嵌套循环、重复计算        |
| 接口是否有分页/缓存    | 避免一次性返回大批数据                  |
| 是否使用异步/并发优化  | 如有大量 I/O 操作时考虑 async、线程池等 |
| 是否存在大量IF条件情况 | 相邻IF条件是否可以合并                  |

---

## 六、可测试性（Testability）

| 检查项                  | 说明                                  |
| ----------------------- | ------------------------------------- |
| 是否覆盖边界/异常场景   | 不仅是 Happy Path，还包括异常处理逻辑 |
| 是否有单元测试/接口测试 | 覆盖新增模块和逻辑分支                |
| 是否破坏已有测试        | 保证老功能不被改坏，CI 能跑通         |
| 是否支持 Mock           | 是否便于模拟依赖环境或数据源          |

---

## 七、兼容性与回滚支持

| 检查项             | 说明                               |
| ------------------ | ---------------------------------- |
| 是否兼容旧逻辑     | 改动是否影响老接口或其他逻辑功能   |
| 是否便于回滚       | 避免一次性多模块修改，影响范围太广 |
| 配置变更是否热更新 | 或支持配置中心管理，避免硬编码     |

---

## 八、文档与其他

| 检查项                | 说明                                        |
| --------------------- | ------------------------------------------- |
| 代码是否易于维护      | 是否为未来迭代预留空间（如配置、参数）      |
| 配置/依赖变更是否记录 | 新增依赖是否在 requirements/pom.yaml 中说明 |
| 是否影响监控/报警     | 是否新增监控点、打点或埋点信息需要同步      |
