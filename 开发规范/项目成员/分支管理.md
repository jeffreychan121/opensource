# GIT分支管理



## 1.  分支定义

- main分支
  - 主分支（github/gitlab原来叫master分支，后面改叫main分支）。
  - 在创建仓库后，main分支会被自动创建。
  - 只有项目的Maintainer有权限修改。
  - 此分支的代码，和生产环境相对应。
- release分支
  - 发布分支。
  - 在测试环境测试通过后，将代码放到对应的`release/<$version-numbber>`分支。
  - 只有项目的Maintainer有权限修改。
  - 此分支的代码，和预发环境（UAT）相对应。

- dev分支
  - 开发分支。
  - 用于开发阶段的集成、测试，包含所有开发中的功能和修复。
  - 只有项目的Maintainer有权限修改。
  - 此分支的代码，和测试环境相对应。
- feature分支
  - 每个新功能，可单独拉一个feature分支。
  - feature的内容，在开发完成并研发自测通过后，提交Merge Request，由项目负责人进行 Review，合并到dev分支。
  - developer有权限修改。
  - 此分支的代码，和研发自己（本地）的环境相对应。
- bug分支
  - 为修复bug，可拉一个单独的bug分支
  - bug分支和feature分支类似。在开发完成并研发自测通过后，提交Merge Request，由项目负责人进行 Review，合并到dev分支。
  - developer有权限修改。
  - 此分支的代码，和研发自己（本地）的环境相对应。


## 2. 分支命名约定

- `main`：生产环境分支，分支名称固定为“main”。
- `release/<version-number>`：用于发布准备的分支。格式为 “feature/” + 版本号，如：
  - release/1.2.0			为 1.2.0 版本发布准备的分支
- `dev`：开发分支，分支名称固定为“dev”。
- `feature/<feature-name>`：新功能分支。名称以“feature/”开头，如：
  - feature/zhangsan-add-change-password-function    张三为“增加修改密码”功能拉的分支    
  - feature/zhangsan-sprint3                          张三为sprint3拉的分支（不太好的做法）
  - feature/zhangsan-requirement-201        张三为 201 这个需求拉的分支

- `bugfix/<bug-name>`：bug修复分支。名称以“bugfix/”开头，如：
  - bugfix/bug-1001		为修复 1001 这个 bug 拉的分支



## 3. 开发阶段

- 首先从main分支，创建dev分支。这个动作是一次性的。
- 从`dev`分支创建一个新feature功能或bug修复分支。
- 在feature功能或bug修复分支上进行开发和测试。使用的是研发自己的本地测试环境。
- 开发完成后，提交并合并（Merge Request）到`dev`分支。
- 项目负责人对MR进行Review，通过后，将代码合并到dev分支。
- 将`dev`分支的内容，部署到测试环境，进行集成测试。

## 4. 发布阶段

- `dev`分支上的代码，部署到测试环境，进行集成测试。当测试通过后，准备好发布时，创建一个`release`分支（如`release/1.0.0`）。

- 将`release`分支的内容，部署到预发环境，进行测试。
  - 如果有测试有bug，则研发需要在feature分支上修改bug，走开发测试流程。

- 预发环境测试通过后，将`release`分支合并到`main`分支。
- 给`main`分支打上对应的标签（tag），如`1.0.0`。
- 将该tag的代码，部署到生产环境。注意，必须使用tag的代码，而不是main分支的代码，去部署生产环境。

## 5. tag管理

每次的版本发布，必须打相应的标签（tag）。标签从`main`分支打。这有助于回溯和查看每个版本的状态。

## 6. 代码审查和合并请求

在合并代码到`dev`或`main`分支前，通过Pull Request或Merge Request进行代码审查。
